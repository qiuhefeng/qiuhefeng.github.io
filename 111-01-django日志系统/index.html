django日志系统

发表于 2019-10-15 更新于 2020-09-07 分类于 100-计算机语言 ， 111-django 阅读次数： Valine：

Django的日志系统使用的是Python自带的标准logging模块。Django内建了几个logging组件方便使用。
本文只介绍了Django扩展的logging功能，logging的使用可以参见另外一篇博客。
第一部分主要介绍如何关闭Django的日志配置。

第三部分主要介绍Django的默认日志配置。
第四部分举例说明，并扩展举例如何结合middleware实现复杂日志记录。

Django Logging扩展

Django内建了自己的Logger、Handlers、Filters。

Loggers

  • django
    django的根logger。

  • django.request
    记录所有请求日志。5xx响应作为ERROR级别。4xx响应作为WARNING级别。
    记录到django.security的请求不会记录到django.request。
    发送给该记录器的消息具有以下额外的上下文：

    • status_code：与请求关联的HTTP响应代码。
    • request：生成日志消息的请求对象。

  • django.server
    与runserver命令调用的服务器接收的请求处理相关的日志消息。HTTP5XX响应记录为ERROR消息，4XX响应记录为WARNING消息，其他所有记录为INFO。
    发送给该记录器的消息具有以下额外的上下文：

    • status_code：与请求关联的HTTP响应代码。
    • request：生成日志消息的请求对象。

  • django.template
    记录与template渲染有关的消息

  • django.db.backends
    记录与数据库相关交互的消息。前提是DEBUG=True。
    发送给该记录器的消息具有以下额外的上下文：

    • duration：执行SQL语句所花费的时间。
    • sql：执行的SQL语句。
    • params：SQL调用中使用的参数。

  • django.security.*
    和安全相关的日志记录器。

  • django.db.backends.shema
    记录数据库操作日志，只记录表结构变化，也就是与migrations famework相关的日志。普通的查询是不会记录在这里的。

Handlers

  • class AdminEmailHandler(include_html=False, email_backend=None)

Django Logging默认配置

  • DEBUG=True
    日志会发往django这个logger（django.server除外），INFO或者更高等级的日志都输出到控制台。
  • DEBUG=False
    日志会发往django这个logger（django.server除外），ERROR或者CRITICAL日志都传递到AdminEmailHandler处理。
  • 其他
    不论DEBUG是否是True，django.server都会把INFO或者更高等级的日志都输出到控制台。

扩展：如何获取Django请求完整日志

Django内建的Logging的配置，一般都能满足基本的使用需求。
实际使用当中可能存在一些需求。比如，出于审计或者其他原因，需要记录每次接口调用的请求内容和返回内容。这个时候需要结合Django的MiddleWare。
Django MiddlerWare简单来说，在每次HTTP请求到达View function之前，都先要经过MiddleWare。返回的时候也是这样。

HttpRequest –> MiddlerWare –> View Function –> MiddlerWare –> HttpResponse

思路： 可以自定义一个MiddleWare，在MiddleWare中进行日志输出，在settings中配置日志行为。
