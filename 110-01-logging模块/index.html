logging模块

发表于 2019-10-10 更新于 2020-10-11 分类于 100-计算机语言 ， 110-python 阅读次数： Valine：

本文介绍Python的标准模块logging。

logging是python标准库提供的记录日志的API

第一部分主要介绍logging的组成部分，包括日志级别、formatter、handler等。第二部分主要介绍logging中各个组件的流转过程，以及logging中层级结构。第三部分主要介绍logging的实际使用。

日志级别

logging默认将日志划分为不同的级别，相当于将日志进行了分类。

级别 数值 描述
CRITICAL 50 严重的错误，程序已经无法运行
ERROR 40 一些严重的问题，程序的某些功能无法使用了
WARNING 30 表示有些预期之外的情况发生，或者在将来可能发生什么情况。程序依然能按照预期运行。
INFO 20 程序正常运行时输出的信息
DEBUG 10 用于调试的日志信息
NOTSET 0 不设置，以父logger级别显示日志

注意：

  • 数值越小，级别越低。也就是说，越低级别的日志包含的信息越多。
  • 当指定输出DEBUG级别的日志时，相对的，大于DEBUG数值的日志也会被输出，不被排除。

logging组成

logging有几种类型组成：

对象 名称 描述
Logger 日志入口 一个日志对象，包含了logging相关的配置信息；同时也是写日志的入口
LogRecord 日志记录器 将日志传到相应的处理器处理
Handler 处理器 将LogRecord生成的记录发送到目的地
Filter 过滤器 对日志进行过滤处理，提供了更好的颗粒度控制
Formatter 格式化器 日志输出格式

Formater格式器

属性名称 使用格式 描述
asctime %(asctime)s 日志事件发生的时间–人类可读时间，如：2003-07-08 16:49:45,896
created %(created)f 日志事件发生的时间–时间戳，就是当时调用time.time()函数返回的值
relativeCreated %(relativeCreated)d 日志事件发生的时间相对于logging模块加载时间的相对毫秒数（目前还不知道干嘛用的）
msecs %(msecs)d 日志事件发生事件的毫秒部分
levelname %(levelname)s 该日志记录的文字形式的日志级别（’DEBUG’, ‘INFO’, ‘WARNING’, ‘ERROR’, ‘CRITICAL’）
levelno %(levelno)s 该日志记录的数字形式的日志级别（10, 20, 30, 40, 50）
name %(name)s 所使用的日志器名称，默认是’root’，因为默认使用的是 rootLogger
message %(message)s 日志记录的文本内容，通过 msg % args计算得到的
pathname %(pathname)s 调用日志记录函数的源码文件的全路径
filename %(filename)s pathname的文件名部分，包含文件后缀
module %(module)s filename的名称部分，不包含后缀
lineno %(lineno)d 调用日志记录函数的源代码所在的行号
funcName %(funcName)s 调用日志记录函数的函数名
process %(process)d 进程ID
processName %(processName)s 进程名称，Python 3.1新增
thread %(thread)d 线程ID
threadName %(thread)s 线程名称

具体如何使用，后面会讲到。

Handler处理器

名称 描述
StreamHandler 流输出。例如输出到屏幕、sys.stdout、sys.stderr 、any file-like object
FileHandler 输出到本地文件
NullHandler 什么都不输出
RotatingFileHandler 输出到本地文件，支持按文件大小切割
TimedRotatingFileHandler 输出到本地文件，支持按时间切割
SocketHandler 输出到套接字，基于TCP
DatagramHandler 输出到套接字，基于UDP
SMTPHandler 将日志通过SMTP发送邮件
HTTPHandler 输出到HTTP
QueueHandler 发送日志到队列

更多支持，查看python文档

logging简要流程

logging处理日志的简要流程：

  • 日志通过Logger写入，再交个Handler处理写到File（日志文件）或者Stream（屏幕、redis等数据流中）
  • 在Logger、Handler中可以通过Filter对日志进行过滤。
  • 在Handler中也可以指定Formatter对日志内容进行格式化处理，可以添加一些logging自带的信息。

logging详细流程

logging处理日志的详细流程：

基于这个流程图，需要说明下Logger的层级结构：

  • Logger的名称是以.进行分割的.例如django.request。那么，request有个父级loggerdjango。
  • Logger有一个默认的顶级父Logger，名字是root。
  • 详细流程图中，当pass to handlers of current logger的同时，会判断是否需要传递到父级的Logger中。如果传递，则继续会传递到父级的handlers处理。是否需要传递，可以通过配置参数propagate来控制（默认False）。
  • 进行日志级别判断时，如果当前Logger没有设置时，会不断寻找父级Logger的日志级别，直到找到。

使用logging

使用logging非常简单，一般过程：

  1. 通过basicConfig()创建一个Logger对象
  2. 调用logging.info()、logging.error()等方法进行写日志

basicCofing()创建Logger对象时，支持传参进行配置。支持如下参数：

参数名称 描述
filename 日志文件名。如果没有，自动会创建。
filemode 文件模式，r[+]、w[+]、a[+]
format 日志输出的格式。详见Formater格式器。默认格式：%(levelname)s:%(name)s%(message)s
datefmt 日志附带日期时间的格式
style 格式占位符，默认为 “%” 和 “{}”
level 设置日志输出级别。默认为WARNING
stream 定义输出流，用来初始化 StreamHandler 对象，不能 filename参数一起使用，否则会ValueError 异常
handles 定义处理器，用来创建 Handler 对象，不能和 filename 、stream参数一起使用，否则也会抛出 ValueError 异常

说明

上述使用logging的方式一般用于单文件的python文件中。
对于多个python文件或者项目，使用单例模式设计，获取Logger对象进行实现。

优点
